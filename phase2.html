Travail 2 Monnayeur et tests
  • Accueil

Objectif

Ce travail consiste à programmer un service pour gérer un tiroir-caisse et le calcul du change nécessaire au paiement d'un achat.

Vous devrez programmer les éléments demandés mais également mettre en place des tests automatiques afin de démontrer la qualité de votre code.

Vous devrez définir des classes d'exception significatives pour les problèmes possibles. Vous les regrouperez dans le package exceptions (1 point).

Fichiers de départ

Vous trouverez un projet Maven de départ ici. Vous pouvez télécharger l'ensemble du code du cours ici

Nom du package et des classes

  • le package org.mon_nom par org.votrenom
  • la même chose pour les classes du package impl

Implémentation de Change

Constructeur

Le constructeur sans paramètre de votre classe doit renvoyer un change vide.

Fonctionnalités

1 point code Le change maintient le nombre d'objets de chaque type, le montant total et le nombre total d'objets après chaque opération.

  • 1 point test on teste les différentes valeurs du change pour un change vide.
  • 1 point test on teste l'évolution des différentes valeurs après des opération d'ajout réussies.
  • 1 point test on teste les différentes exceptions sur des opérations avec exceptions

Implémentation de ServiceArgent

Constructeur

Le constructeur sans paramètre renvoit un service qui est à moitié plein. Par exemple, il contient 20 billets de chaque sorte puisque la capacité maximale pour les billets et les pièces est de 40 par type.

Calcul de l'arrondi

1 point code 3 points test Vous devez vous assurer que votre calcul d'arrondi est correct dans tous les cas. Tous les arrondis sont testés .01 .02 .03 .04 etc .09 (source)

Calcul de la monnaie simple

1 point code 2 points test Il existe un test de base sur le calcul où un montant simple est retiré avec un change suffisant et que le Service est capable de contenir. Les montants négatifs sont testés ainsi que les autres montants aberrants, une exception significative est levée.

Calcul de la monnaie avec dépassement

1 point code 1 point test Il existe un test illustrant un cas où l'argent donné ne rentre pas dans les capacités du service. Par exemple, on paie avec 3 billets de 100$ alors que le service contient déjà 38 billets de 100$. Une exception doit être lancé et l'état du service ne doit pas changer.

Calcul de la monnaie, manque de fonds

1 point code 2 points test Votre service doit gérer les cas où le change ne peut pas être rendu parce qu'il manque quelque chose dans le service. Dans ces situations, le service doit rester inchangé. Par exemple, on veut le change pour 60$ et il ne reste que 20 dans le service.

Calcul de la monnaie, manque certains objets

1 point code 2 points test Vous devez gérer des cas comme le suivant : il ne reste que des billets (30 de chaque type) et on doit rendre le change pour 63$. Cela devrait échouer (le client nous devait 37$ et il a payé avec un billet de 100$).

2 points code (difficile) 1 point test Vous devez gérer le type de cas suivant : il reste un billet des billets de 50$ et des billets de 20$ mais rien d'autre. Quelqu'un demande un change de 60$ (il doit 40$ et il a donné 100$), Cela devrait réussir avec un résultat de 3 * 20$).

Autres méthodes du service

2 points code 4 points test Vous devez tester les autres méthodes du service (hors arrondi et calcul du change) et vous assurer de leur bon fonctionnement.

Couverture de code

4 points test Regroupez tout le code que vous avez produit dans le package impl pour faciliter le calcul de la couverture de code (pourcentage des lignes de code).

  • 1 point entre 50% et 75%
  • 2 points entre 75% et 85%
  • 3 points entre 85% et 95%
  • 4 points > 95%

Manipulation par les interfaces

4 points négatifs Tous vos traitements doivent utiliser les méthodes définies dans les interfaces. Vous devez manipuler les objets par les interfaces Change et ServiceArgent plutôt que par vos classes d'implémentations. Vous perdrez un point à chaque fois que :

  • Vous accédez directement à un champ d'une classe depuis une autre classe (dans le code ou dans les tests)
  • Vous utilisez une méthode définie dans vos classes mais qui n'existe pas dans l'interface (dans le code ou dans les tests)
