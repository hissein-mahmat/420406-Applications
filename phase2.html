Travail 2 Monnayeur et tests
  • Accueil

Objectif

Ce travail consiste à programmer un service pour gérer un tiroir-caisse et le calcul du change nécessaire au paiement d'un achat.

Vous devrez programmer les éléments demandés mais également mettre en place des tests automatiques afin de démontrer la qualité de votre code.

Vous devrez définir des classes d'exception significatives pour les problèmes possibles. Vous les regrouperez dans un package exceptions (1 point).

Implémentations de Change et TiroirCaisse

Nom du package et des classes

1 point Vous devez renommer:

  • le package org.mon_nom par org.votrenom
  • la même chose pour les 3 classes du package impl
Fonctionnalités

1 point code Le tiroir caisse et le change maintiennent le nombre d'objets de chaque type, le montant total et le nombre total d'objets après chaque opération.

  • 1 point test on teste le dépassement de la capacité pour un type d'objet à l'ajout d'objets pour un tiroir caisse. Pour chaque billet, il y a 40 emplacements. Pour chaque pièce, il y a 50 emplacements.
  • 1 point test on teste le tiroir-caisse plein au constructeur : le compartiment des pièces de 1 sou doit être vide puisqu'on ne rend jamais de 1 sou. Tous les autres compartiments sont pleins à leur capacité.
  • 1 point test on teste le manque d'objets quand on retire trop d'objets d'un certain type pour un tiroir-caisse
  • 1 point test on teste l'impossibilité de retirer des 1 sou. (par contre on peut en ajouter)
Constructeur

Le constructeur par défaut (sans paramètres) de votre classe doit renvoyer une tiroir plein (sauf les 1 sou).

Le montant total est de 40 billets de 100, 40 billets de 50 etc. Le nombre total d'objets est de 40 + 40 + 40 ... 50 + 50 + 50 etc.

Implémentation de ServiceArgent

Calcul de l'arrondi

1 point code 3 points test Vous devez vous assurer que votre calcul d'arrondi est correct dans tous les cas. Tous les arrondis sont testés .01 .02 .03 .04 etc .09 (source)

Calcul de la monnaie simple

1 point code 2 points test Il existe un test de base sur le calcul. Les montants négatifs sont testés ainsi que les autres montants aberrants, une exception significative est levée.

Calcul de la monnaie, manque de fonds

1 point code 2 points test Votre service doit gérer les cas où le montant ne peut pas être rendu parce qu'il manque quelque chose dans le tiroir-caisse. Dans ces situations, le tiroir-caisse doit rester inchangé. Par exemple, on veut le change pour 60$ et il ne reste que 20 dans le tiroir-caisse.

Calcul de la monnaie, manque certains objets

1 point code 2 points test Vous devez gérer des cas comme le suivant : il ne reste que des billets (40 de chaque type) et on veut calculer le change pour 63$. Cela devrait échouer.

2 points code (difficile) 1 point test Vous devez gérer le type de cas suivant : il reste un billet des billets de 50$ et des billets de 20$ dans le tiroir caisse mais rien d'autre. Quelqu'un demande 60$ dans la caisse. Cela devrait réussir (3 billets de 20$).

Couverture de code

4 points test Regroupez tout le code que vous avez produit dans le package impl pour faciliter le calcul de la couverture de code.

  • 1 point entre 50% et 75%
  • 2 points entre 75% et 85%
  • 3 points entre 85% et 95%
  • 4 points > 95%

Manipulation par les interfaces

4 points négatifs Tous vos traitements doivent utiliser les méthodes définies dans les interfaces. Vous devez manipuler les objets par les interfaces Change, TiroirCaisse et ServiceArgent plutôt que par vos classes d'implémentations. Vous perdrez un point à chaque fois que :

  • Vous accédez directement à un champ d'une classe depuis une autre classe (dans le code ou dans les tests)
  • Vous utilisez une méthode définie dans vos classes mais qui n'existe pas dans l'interface (dans le code ou dans les tests)
